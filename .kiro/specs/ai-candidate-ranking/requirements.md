# Requirements Document

## Introduction

This document specifies the requirements for a candidate ranking system built for the **Redrob Intelligent Candidate Discovery & Ranking Challenge**. The system reads a pool of up to 100,000 candidate profiles and a single fixed job description (Senior AI Engineer — Founding Team at Redrob AI), reasons about genuine job fit, and produces a ranked top-100 shortlist as a CSV submission file with grounded natural-language reasoning for each candidate.

The defining characteristic of the system is that the ranking step runs **fully local, CPU-only, and offline**. During ranking there is no network access of any kind: no calls to hosted LLM or embedding services (OpenAI, Anthropic, Cohere, Gemini, Bedrock, or similar) and no other external API calls. All intelligence is provided by locally-hosted models (for example sentence-transformers BGE/E5 embedding models and a FAISS vector index) and by structured feature logic. Heavy work such as generating embeddings or building indexes may be pre-computed offline, but any pre-computed artifact must be committed to the repository or produced by a script, and the ranking command that writes the submission CSV must complete within a strict compute budget without network access.

Ranking quality is judged on genuine fit rather than keyword counting. The system must reason about the gap between what the job description literally says and what it actually requires (production embeddings, retrieval, and ranking experience at product companies; strong Python; evaluation-framework experience such as NDCG/MRR/MAP). It must reward strong-but-unbuzzwordy candidates, down-weight keyword stuffers and other negative-signal profiles, incorporate behavioral availability signals, and detect inconsistent "honeypot" profiles so they do not appear in the top 100.

Submissions are scored against a hidden ground truth using a weighted blend of ranking-quality metrics: NDCG@10 (weight 0.50), NDCG@50 (weight 0.30), MAP (weight 0.15), and Precision@10 (weight 0.05). Because more than two thirds of the weight falls on the top 10 and top 50 positions, the quality and ordering of the highest-ranked candidates matters most. The challenge also provides a `sample_submission.csv` that is a deliberate anti-pattern: it ranks keyword-stuffers (for example an "HR Manager with 9 AI core skills") at the top, exactly the outcome this system must avoid.

## Glossary

- **Ranking_Pipeline**: The end-to-end local system that ingests candidates, scores them, ranks them, generates reasoning, and writes the Submission_CSV. Invoked by a single reproduce command.
- **Candidate_Record**: A single candidate profile from the input dataset, conforming to the Redrob Candidate Profile Schema, identified by a Candidate_Id.
- **Candidate_Id**: A unique candidate identifier string matching the pattern `^CAND_[0-9]{7}$`.
- **Candidate_Pool**: The complete set of Candidate_Records loaded from the input dataset (up to 100,000 records).
- **Job_Description**: The single fixed role document for "Senior AI Engineer — Founding Team at Redrob AI", supplied as `job_description.md` / `.docx`.
- **Job_Profile**: The structured, offline-derived representation of the Job_Description requirements (true signals, disqualifiers, location and notice-period preferences) used for scoring.
- **Redrob_Signals**: The `redrob_signals` object on each Candidate_Record containing 23 behavioral and engagement fields (e.g. `profile_completeness_score`, `last_active_date`, `recruiter_response_rate`, `github_activity_score`, `notice_period_days`).
- **Embedding_Model**: A locally-hosted text embedding model (e.g. a sentence-transformers BGE or E5 model) that runs on CPU with no network access.
- **Semantic_Embedding**: A numeric vector produced by the Embedding_Model representing the meaning of a text (a Candidate_Record summary or the Job_Profile).
- **Vector_Index**: A local similarity-search index (e.g. FAISS) over Candidate Semantic_Embeddings used to compute semantic fit efficiently.
- **Precomputed_Artifact**: Any artifact generated by offline pre-computation (Semantic_Embeddings, Vector_Index, model weights) that is committed to the repository or produced by a documented build script.
- **Fit_Score**: A numeric measure of how well a Candidate_Record genuinely matches the Job_Profile, combining semantic similarity and structured feature signals before behavioral modification.
- **Behavioral_Modifier**: A multiplier or adjustment derived from Redrob_Signals (activity, responsiveness, availability) applied on top of the Fit_Score.
- **Final_Score**: The score used for ranking, produced by applying the Behavioral_Modifier to the Fit_Score; written to the `score` column of the Submission_CSV.
- **Honeypot**: A Candidate_Record containing internally inconsistent or impossible data (e.g. years_of_experience exceeding the candidate's company founding window, or "expert" proficiency with zero duration_months) planted to penalize naive rankers.
- **Keyword_Stuffer**: A Candidate_Record that lists many job-relevant skills or terms but whose title and career history show no genuine relevant production experience.
- **Consistency_Check**: A plausibility validation applied to a Candidate_Record to detect Honeypot or internally inconsistent data without referencing hardcoded candidate identifiers.
- **Ranked_Shortlist**: The ordered list of the top 100 Candidate_Records sorted by Final_Score (descending) with deterministic tie-breaking.
- **Reasoning**: A 1-2 sentence, grounded, non-templated justification per ranked candidate written to the `reasoning` column of the Submission_CSV.
- **Submission_CSV**: The output file with header `candidate_id,rank,score,reasoning` and exactly 100 data rows, conforming to the challenge auto-validator.
- **Reproduce_Command**: The single command `python rank.py --candidates ./candidates.jsonl --out ./submission.csv` that runs the Ranking_Pipeline end-to-end and writes the Submission_CSV.
- **Compute_Budget**: The hard runtime constraints for the ranking step: completion within 5 minutes wall-clock for 100,000 candidates, CPU only (no GPU), at most 16 GB RAM, at most 5 GB intermediate disk state, and no network access.
- **Scoring_Config**: The configurable set of weights and parameters controlling how scoring dimensions combine into the Fit_Score and Final_Score.
- **Deliverables**: The challenge submission artifacts: the GitHub repository, the Submission_CSV, the presentation deck (PDF), the `submission_metadata.yaml` file, and the sandbox/demo link.
- **Evaluation_Metrics**: The hidden ground-truth scoring blend used to judge a Submission_CSV: NDCG@10 (weight 0.50), NDCG@50 (weight 0.30), MAP (weight 0.15), and Precision@10 (weight 0.05).

## Requirements

### Requirement 1: Dataset Ingestion

**User Story:** As a participant, I want the system to load the full candidate dataset from the challenge files, so that every candidate in the pool is available for ranking.

#### Acceptance Criteria

1. WHEN a path to a JSON Lines file (`candidates.jsonl`) is provided, THE Ranking_Pipeline SHALL parse each line as one Candidate_Record and load all valid records into the Candidate_Pool.
2. WHEN the provided dataset path references a gzip-compressed file (`candidates.jsonl.gz`), THE Ranking_Pipeline SHALL decompress and parse the file without requiring a separate manual decompression step.
3. WHEN the provided dataset is a JSON array sample file (`sample_candidates.json`) containing at most 100 records, THE Ranking_Pipeline SHALL parse and load all records into the Candidate_Pool.
4. THE Ranking_Pipeline SHALL load a Candidate_Pool of up to 100,000 Candidate_Records within the Compute_Budget.
5. IF a line or record fails to parse as valid JSON or is missing a Candidate_Id matching `^CAND_[0-9]{7}$`, THEN THE Ranking_Pipeline SHALL skip that record, record a warning identifying the line number and the rejection reason, and continue loading remaining records.
6. IF the dataset file cannot be read (file not found, permission denied, or unreadable), THEN THE Ranking_Pipeline SHALL return an error identifying the specific access failure.
7. IF the Candidate_Pool contains zero valid Candidate_Records after loading, THEN THE Ranking_Pipeline SHALL return an error indicating that no valid candidates were found.

### Requirement 2: Job Role Understanding (Offline)

**User Story:** As a participant, I want the system to interpret the job description offline, so that scoring reflects the true requirements of the role without any network calls.

#### Acceptance Criteria

1. WHEN the Ranking_Pipeline starts, THE Ranking_Pipeline SHALL derive a Job_Profile for the Senior AI Engineer role using only local resources and no network access.
2. THE Job_Profile SHALL represent the role's true positive signals, including production embeddings, retrieval, and ranking experience at product companies, strong Python proficiency, and evaluation-framework experience (NDCG, MRR, or MAP).
3. THE Job_Profile SHALL represent the role's negative signals, including keyword-stuffer titles unrelated to engineering, pure-research or academic backgrounds without production experience, AI experience limited to recent (under 12 months) LangChain-calling-OpenAI work, seniors who have not coded in 18 or more months, title-chasing job-hopping patterns (for example role changes roughly every 1.5 years for title bumps), careers spent entirely at consulting or services firms (for example TCS, Infosys, Wipro, Accenture, Cognizant, Capgemini), primary expertise in computer vision, speech, or robotics without NLP or information retrieval, and closed-source-only work for 5 or more years with no external validation.
4. THE Job_Profile SHALL represent the role's location preference (Noida and Pune as preferred Tier-1 Indian cities with relocation acceptable, locations outside India evaluated case-by-case) and notice-period preference (under 30 days preferred, 30 or more days raising the bar).
5. WHEN a Semantic_Embedding of the Job_Profile is required, THE Ranking_Pipeline SHALL generate it using the local Embedding_Model with no network access.

### Requirement 3: Candidate Fit Scoring

**User Story:** As a participant, I want each candidate scored on genuine fit, so that ranking reflects real capability rather than keyword presence.

#### Acceptance Criteria

1. WHEN scoring a Candidate_Record, THE Ranking_Pipeline SHALL compute a Fit_Score that combines semantic similarity to the Job_Profile with structured feature signals derived from the candidate's profile, career_history, education, skills, and certifications.
2. WHEN computing semantic similarity, THE Ranking_Pipeline SHALL use cosine similarity between the candidate's Semantic_Embedding and the Job_Profile Semantic_Embedding, producing a value between 0.0 and 1.0.
3. WHERE a Candidate_Record's career_history shows building a recommendation, search, or ranking system at a product company, THE Ranking_Pipeline SHALL treat that career_history as strong evidence of fit even when the record contains no AI buzzwords in its skills or headline.
4. WHEN a Candidate_Record exhibits one or more negative signals defined in the Job_Profile, THE Ranking_Pipeline SHALL reduce that candidate's Fit_Score relative to an otherwise equivalent candidate without those negative signals.
5. THE Ranking_Pipeline SHALL produce a Fit_Score for every Candidate_Record in the Candidate_Pool that passes ingestion.
6. THE Ranking_Pipeline SHALL compute Fit_Scores deterministically, such that two runs over the same Candidate_Pool and Scoring_Config produce identical Fit_Scores.

### Requirement 4: Keyword-Stuffer and Trap Resistance

**User Story:** As a participant, I want the system to resist keyword stuffing, so that profiles padded with relevant terms but lacking real experience do not rank highly.

#### Acceptance Criteria

1. WHEN a Candidate_Record lists many job-relevant skills but the current_title and career_history titles are unrelated to AI or software engineering (for example "Marketing Manager"), THE Ranking_Pipeline SHALL classify the record as a Keyword_Stuffer and reduce its Fit_Score below that of candidates with genuine relevant titles and history.
2. WHEN a skill on a Candidate_Record is marked "expert" or "advanced" proficiency but has a `duration_months` of 0, THE Ranking_Pipeline SHALL discount that skill's contribution to the Fit_Score.
3. THE Ranking_Pipeline SHALL weight evidence from current_title and career_history more heavily than the raw count of job-relevant skills listed when computing the Fit_Score.
4. THE Ranking_Pipeline SHALL classify Keyword_Stuffers using profile and career evidence only, without referencing hardcoded candidate identifiers.

### Requirement 5: Behavioral Signal Modifier

**User Story:** As a participant, I want behavioral signals to adjust scores, so that strong-on-paper but unavailable or unresponsive candidates are ranked appropriately lower.

#### Acceptance Criteria

1. WHEN computing the Final_Score, THE Ranking_Pipeline SHALL apply a Behavioral_Modifier derived from the candidate's Redrob_Signals to the Fit_Score.
2. WHILE a Candidate_Record's `last_active_date` indicates inactivity of 6 or more months relative to the most recent activity date in the Candidate_Pool, THE Ranking_Pipeline SHALL reduce that candidate's Final_Score through the Behavioral_Modifier.
3. WHEN a Candidate_Record has a low `recruiter_response_rate`, THE Ranking_Pipeline SHALL reduce that candidate's Final_Score through the Behavioral_Modifier relative to an otherwise equivalent candidate with a higher response rate.
4. THE Behavioral_Modifier SHALL incorporate availability and engagement signals including `last_active_date`, `recruiter_response_rate`, `notice_period_days`, and `open_to_work_flag`.
5. THE Ranking_Pipeline SHALL apply the Behavioral_Modifier such that it adjusts but does not solely determine the Final_Score, preserving the relative influence of the Fit_Score.

### Requirement 6: Honeypot and Inconsistency Detection

**User Story:** As a participant, I want the system to detect inconsistent profiles, so that honeypot candidates do not appear in the top 100 and cause disqualification.

#### Acceptance Criteria

1. WHEN scoring a Candidate_Record, THE Ranking_Pipeline SHALL apply Consistency_Checks that detect internal contradictions between the profile, career_history dates, skills, and Redrob_Signals.
2. IF a Candidate_Record's `years_of_experience` exceeds the time elapsed since the earliest plausible founding or start window implied by the candidate's career_history, THEN THE Ranking_Pipeline SHALL flag the record as a Honeypot.
3. IF a Candidate_Record claims "expert" proficiency in multiple skills each with `duration_months` of 0, THEN THE Ranking_Pipeline SHALL flag the record as a Honeypot.
4. WHEN a Candidate_Record is flagged as a Honeypot, THE Ranking_Pipeline SHALL exclude or down-weight that record so that flagged records do not enter the Ranked_Shortlist except where unavoidable.
5. THE Ranked_Shortlist SHALL contain fewer than 10 percent Honeypot records, with a target of zero, since a top 100 containing 10 percent or more Honeypot records causes disqualification.
6. THE Ranking_Pipeline SHALL detect Honeypots using Consistency_Checks on profile data only, without referencing hardcoded candidate identifiers.

### Requirement 7: Ranking and Deterministic Tie-Breaking

**User Story:** As a participant, I want a deterministic ranked top-100, so that the output is reproducible and conforms to the submission rules.

#### Acceptance Criteria

1. WHEN all Final_Scores are computed, THE Ranking_Pipeline SHALL sort Candidate_Records in descending order of Final_Score and select the top 100 to form the Ranked_Shortlist.
2. THE Ranking_Pipeline SHALL assign each record in the Ranked_Shortlist an integer rank from 1 to 100, where rank 1 is the best fit and each rank value appears exactly once.
3. THE Ranking_Pipeline SHALL ensure the Final_Score is monotonically non-increasing as rank increases, such that the score at rank n is greater than or equal to the score at rank n+1.
4. WHEN two or more Candidate_Records have equal Final_Scores, THE Ranking_Pipeline SHALL break the tie deterministically by ordering those records by Candidate_Id ascending.
5. THE Ranking_Pipeline SHALL produce identical Ranked_Shortlist output across repeated runs over the same Candidate_Pool and Scoring_Config.

### Requirement 8: Reasoning Generation (Offline, Grounded, Varied)

**User Story:** As a participant, I want grounded reasoning for each ranked candidate, so that the justification is trustworthy and explains the fit.

#### Acceptance Criteria

1. WHEN a Candidate_Record is placed in the Ranked_Shortlist, THE Ranking_Pipeline SHALL generate a Reasoning of 1 to 2 sentences for that record using only local resources and no network access.
2. THE Reasoning SHALL reference specific facts drawn from the candidate's own Candidate_Record, including at least one of: years of experience, title, a named skill, or a specific Redrob_Signals value.
3. THE Reasoning SHALL connect the referenced candidate facts to specific Job_Profile requirements.
4. WHEN a Candidate_Record has a notable concern or gap relative to the Job_Profile, THE Reasoning SHALL acknowledge that concern or gap.
5. THE Reasoning SHALL reference only skills, employers, and attributes present in the candidate's Candidate_Record, with no hallucinated facts.
6. THE Reasoning text SHALL be substantively varied across the Ranked_Shortlist rather than templated, and its tone SHALL be consistent with the candidate's rank.

### Requirement 9: Submission CSV Output and Validation Conformance

**User Story:** As a participant, I want the output to pass the challenge auto-validator, so that my submission is accepted.

#### Acceptance Criteria

1. WHEN ranking is complete, THE Ranking_Pipeline SHALL write the Ranked_Shortlist to the output path as a UTF-8 encoded CSV file with a `.csv` extension and no other output format.
2. THE Submission_CSV SHALL have as its first row exactly the header `candidate_id,rank,score,reasoning`.
3. THE Submission_CSV SHALL contain exactly 100 data rows following the header, one per ranked candidate.
4. THE Submission_CSV SHALL write for each data row the Candidate_Id, the integer rank, the float Final_Score, and the Reasoning, in that column order.
5. THE Submission_CSV SHALL contain only Candidate_Ids that exist in the Candidate_Pool and match `^CAND_[0-9]{7}$`, with no duplicate Candidate_Id and no duplicate rank.
6. THE Submission_CSV SHALL order rows such that the `score` column is monotonically non-increasing as `rank` increases, and equal scores are ordered by Candidate_Id ascending.
7. IF the output file cannot be written (permission denied or insufficient disk space), THEN THE Ranking_Pipeline SHALL return an error describing the failure reason.

### Requirement 10: Compute-Constraint Conformance

**User Story:** As a participant, I want the ranking step to respect the hard compute limits, so that the submission is not disqualified at review.

#### Acceptance Criteria

1. WHEN the Reproduce_Command is run over a 100,000-candidate Candidate_Pool, THE Ranking_Pipeline SHALL complete and write the Submission_CSV within 5 minutes of wall-clock time.
2. THE Ranking_Pipeline SHALL perform ranking using CPU only and SHALL NOT require a GPU.
3. THE Ranking_Pipeline SHALL complete ranking using at most 16 GB of RAM.
4. THE Ranking_Pipeline SHALL use at most 5 GB of intermediate disk state during ranking.
5. WHILE the ranking step is running, THE Ranking_Pipeline SHALL make no network requests to any hosted LLM service, hosted embedding service, or other external API.
6. THE Ranking_Pipeline SHALL produce the Submission_CSV using only locally-hosted models and locally-stored Precomputed_Artifacts.

### Requirement 11: Offline Pre-Computation of Artifacts

**User Story:** As a participant, I want to pre-compute heavy artifacts offline, so that the ranking step stays within the Compute_Budget while remaining reproducible.

#### Acceptance Criteria

1. WHERE Semantic_Embeddings or a Vector_Index are pre-computed before ranking, THE Ranking_Pipeline SHALL load those Precomputed_Artifacts from local storage during ranking with no network access.
2. THE repository SHALL either commit each Precomputed_Artifact or provide a documented script that builds each Precomputed_Artifact offline.
3. WHEN pre-computation is used, THE Deliverables SHALL document the pre-computation process and its approximate runtime.
4. THE pre-computation process SHALL use only locally-hosted models and SHALL NOT call any hosted LLM or embedding service.
5. IF an expected Precomputed_Artifact is missing at ranking time, THEN THE Ranking_Pipeline SHALL return an error identifying the missing artifact and how to build it.

### Requirement 12: Configurable Scoring

**User Story:** As a participant, I want to adjust scoring weights, so that I can tune the ranking behavior without changing code.

#### Acceptance Criteria

1. THE Ranking_Pipeline SHALL expose a Scoring_Config controlling the weights of the scoring dimensions that combine into the Fit_Score and the strength of the Behavioral_Modifier.
2. WHEN no Scoring_Config is supplied, THE Ranking_Pipeline SHALL apply documented default weights.
3. WHEN a Scoring_Config supplies custom weights, THE Ranking_Pipeline SHALL apply those weights when computing the Fit_Score and Final_Score.
4. IF a Scoring_Config contains a negative weight, THEN THE Ranking_Pipeline SHALL return a validation error indicating that weights must be non-negative.
5. IF a Scoring_Config sets all weights to zero, THEN THE Ranking_Pipeline SHALL return a validation error indicating that at least one weight must be greater than zero.

### Requirement 13: Deliverables

**User Story:** As a participant, I want all required submission artifacts produced, so that the entry is complete and reviewable.

#### Acceptance Criteria

1. THE Deliverables SHALL include a working GitHub repository containing a README, a dependency specification (`requirements.txt` or `pyproject`), and the Ranking_Pipeline source.
2. THE repository SHALL support the single Reproduce_Command `python rank.py --candidates ./candidates.jsonl --out ./submission.csv` that produces the Submission_CSV.
3. THE repository SHALL include either committed Precomputed_Artifacts or a script that builds them offline.
4. THE repository SHALL include a `submission_metadata.yaml` file at its root that declares team identity and primary contact, the GitHub repository URL, the sandbox link, the Reproduce_Command, the compute environment (including `uses_gpu_for_inference: false` and `has_network_during_ranking: false`), any pre-computation use and its approximate runtime, the AI tools used with a usage summary, a methodology summary, and the required declarations.
5. THE Deliverables SHALL include the ranked top-100 Submission_CSV.
6. THE Deliverables SHALL include a presentation deck explaining the approach, converted to PDF.
7. THE Deliverables SHALL include a hosted sandbox or demo link (HuggingFace Spaces, Streamlit, Replit, Colab, Docker, or Binder) that runs the Ranking_Pipeline on a sample of at most 100 candidates end-to-end within the Compute_Budget.
8. WHERE no hosted sandbox is provided, THE Deliverables SHALL include a self-contained `docker run` recipe in the README that runs the Ranking_Pipeline on a sample of at most 100 candidates end-to-end within the Compute_Budget.
